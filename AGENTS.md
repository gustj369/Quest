# Quest — Habit Tracker RPG Agent Guide

AI 코딩 에이전트가 이 저장소에서 작업할 때 따르는 기준 문서입니다.  
목표는 **픽셀아트 RPG 감성의 일상 습관 트래커 PWA**를 유지하면서, 실제 사용하기 편한 모바일 앱으로 안정적으로 발전시키는 것입니다.

---

## 1. Role

이 저장소에서는 신중한 코딩 에이전트로 행동합니다.

- 관련 파일을 먼저 읽고 기존 구조를 이해합니다.
- 넓은 변경은 이유를 설명한 뒤 진행합니다.
- 작은 단위의 검토 가능한 변경을 선호합니다.
- 기존 아키텍처를 보존하고, 명확한 이유가 있을 때만 바꿉니다.
- 코드 수정 전에는 의도한 변경을 요약합니다.
- 코드 수정 후에는 무엇을 바꿨는지, 어떻게 확인하면 되는지 설명합니다.

---

## 2. 작업 전 반드시 확인

1. **디자인 원칙**
   - 픽셀아트 RPG 감성 + 현대 모바일 앱의 쾌적함을 함께 유지합니다.
   - 둘이 충돌하면 **깔끔함과 사용성**을 우선합니다.
   - 픽셀 감성은 포인트로만 사용하고, 전체 레이아웃은 모바일 기준으로 설계합니다.
   - 제네릭 AI 디자인 금지: 보라 그라디언트 남발, Inter 폰트 기본값, 의미 없는 장식형 카드 UI를 피합니다.

2. **기술 스택**
   - React 18 + Vite 5
   - Tailwind CSS + CSS Variables
   - CSS Transitions 기본, Framer Motion은 레벨업 연출 중심
   - 데이터는 MVP 기준 localStorage, v2에서 Supabase 예정
   - PWA는 vite-plugin-pwa
   - 아이콘은 lucide-react
   - 폰트는 Press Start 2P(픽셀 포인트), Noto Sans KR(본문)

3. **컬러 시스템**
   - 기존 CSS 변수와 색상 의도를 우선 사용합니다.
   - 새 색상이 필요하면 아래 팔레트와 조화되게 추가합니다.

```css
--bg-base: #1e1a2e;
--bg-card: #2a2640;
--accent-primary: #7fdbca;   /* 민트: 완료, 주요 액션 */
--accent-secondary: #f5c542; /* 노랑: XP, 레벨업 */
--accent-danger: #ff6b6b;    /* 레드: 경고, 삭제 */
--text-primary: #f0ece8;
--text-muted: #8a8499;
--pixel-shadow: 3px 3px 0px #000;
```

## 3. Project Context

### 프로젝트 목적

Quest는 사용자가 매일 퀘스트(습관)를 완료하며 캐릭터를 성장시키는 픽셀아트 RPG 감성의 습관 트래커 PWA입니다.

현재는 localStorage 기반 MVP이며, 장기적으로는 Supabase 동기화, 푸시 알림, 퀘스트 수정, 스마트워치 연동을 고려합니다.

### 핵심 폴더

```text
src/
  utils/       nanoid, storage, xp, defaults, date
  hooks/       useQuests, useCharacter
  components/  UI 컴포넌트 (Header, XPBar, QuestCard, ...)
  screens/     HomeScreen, CategoriesScreen, CharacterScreen
ABOUT-ME/      사용자 맥락 참고용
docs/          프로젝트 문서, 용어집, 플랜
TEMPLATES/     새 기능 추가 시 참고 템플릿
OUTPUTS/       결과물 저장소
```

새 파일은 이 구조를 먼저 따르고, 애매하면 기존 코드의 배치 방식을 우선합니다.

### 실행 방법

```bash
npm run dev    # 개발 서버 (localhost:5173)
npm run build  # 프로덕션 빌드 -> dist/
npm run preview
```

### 테스트 방법

현재 별도 테스트 스크립트는 없습니다.  
작업 후에는 최소한 `npm run build`로 빌드 오류를 확인하고, UI 변경은 브라우저에서 모바일 기준 화면을 확인합니다.

## 4. 핵심 도메인

- 퀘스트 CRUD + 원탭 완료
- 완료 시 슬라이드 아웃 애니메이션
- 카테고리 분류 + 탭 전환
- repeat 반복 주기: `daily`, `weekday`, `weekend`, `weekly`
- EXP 시스템 + 레벨업 연출
- 캐릭터 화면: 스프라이트, 통계, 잔디 달력, 뱃지
- 설정 시트: 이름 변경, 알림 토글, 2단계 데이터 리셋
- PWA 홈화면 추가 배너
- ALL CLEAR! 배너

## 5. Coding Style

- **네이밍 규칙**
  - React 컴포넌트는 PascalCase를 사용합니다.
  - 훅은 `useSomething` 형태를 사용합니다.
  - 유틸 함수는 camelCase를 사용합니다.
  - repeat, difficulty 같은 도메인 값은 기존 문자열 계약을 유지합니다.

- **파일 구조 규칙**
  - 화면 단위는 `src/screens/`에 둡니다.
  - 재사용 UI는 `src/components/`에 둡니다.
  - 상태 로직은 `src/hooks/`에 둡니다.
  - 순수 계산, 저장소, 기본값은 `src/utils/`에 둡니다.

- **에러 처리 방식**
  - localStorage 접근은 실패 가능성을 고려합니다.
  - 사용자 데이터 삭제, 초기화, 마이그레이션은 특히 보수적으로 다룹니다.
  - 알림, PWA 설치 등 브라우저 API는 지원 여부와 권한 상태를 확인합니다.

- **주석 기준**
  - 코드만으로 의도가 분명하면 주석을 추가하지 않습니다.
  - 날짜 리셋, XP 계산, 데이터 마이그레이션처럼 실수 비용이 큰 로직에는 짧은 설명을 남깁니다.
  - 깨진 한글 주석이나 문자열을 발견하면 기능 변경과 분리해서 정리합니다.

- **커밋/PR 기준**
  - 작은 변경 단위로 묶습니다.
  - 변경 이유, 확인 방법, 남은 위험을 명확히 적습니다.
  - 불필요한 포맷팅이나 관련 없는 파일 변경을 섞지 않습니다.

## 6. Workflow

1. 관련 파일을 먼저 읽습니다.
2. 이해한 내용을 짧게 요약합니다.
3. 최소 변경 계획을 제안합니다.
4. 작은 단위로 수정합니다.
5. `npm run build` 또는 적절한 확인 방법을 실행합니다.
6. 변경 파일, 확인 결과, 남은 위험을 보고합니다.

---

## 7. 코드 수정 전 체크리스트

- [ ] 요청이 MVP 범위인지, v2 기능인지 구분했습니다.
- [ ] 관련 컴포넌트, 훅, 유틸의 기존 패턴을 확인했습니다.
- [ ] localStorage 데이터 구조에 영향이 있는지 확인했습니다.
- [ ] 디자인 원칙과 컬러 시스템을 확인했습니다.
- [ ] 모바일 화면에서 레이아웃이 무너지지 않을 변경인지 생각했습니다.
- [ ] 변경 범위를 작게 유지할 수 있는지 확인했습니다.

---

## 8. 코드 수정 후 체크리스트

- [ ] 변경 범위가 요청과 직접 관련된 파일에 머무릅니다.
- [ ] 기존 퀘스트 완료, XP, 레벨업, 카테고리 흐름이 깨지지 않습니다.
- [ ] localStorage 기존 데이터와 호환됩니다.
- [ ] UI 텍스트, 버튼, 상태 표시가 한국어 앱 흐름에 자연스럽습니다.
- [ ] `npm run build`를 실행해 빌드 오류를 확인했습니다.
- [ ] 필요하다면 브라우저에서 모바일 기준 화면을 확인했습니다.

---

## 9. Do Not

- 추측으로 대규모 리팩토링하지 않습니다.
- 기존 API 계약이나 localStorage 스키마를 임의로 바꾸지 않습니다.
- 테스트나 빌드 확인 없이 핵심 로직을 변경하지 않습니다.
- 불필요한 의존성을 추가하지 않습니다.
- 디자인 원칙을 무시한 제네릭 UI를 만들지 않습니다.
- 사용자의 기존 데이터를 지우는 변경을 가볍게 처리하지 않습니다.
