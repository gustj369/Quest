# Testing Strategy

Quest의 테스트는 MVP 안정성을 높이는 범위에서 작게 시작합니다.
새 의존성은 실제 필요가 확인될 때만 추가합니다.

## 1. 현재 테스트 범위

현재 `npm.cmd test`는 Node 단일 스크립트로 실행합니다.

```powershell
npm.cmd test
```

포함된 범위:

- localStorage 데이터 normalize
- XP 계산 경계값
- 날짜 key 포맷과 당일 리셋 방어

이 방식은 React 렌더링 없이 실행되므로 빠르고, Codex 세션의 `node --test` spawn 제약을 피합니다.

## 2. 아직 테스트하지 않는 범위

`useQuests` 같은 React hook 자체 테스트는 아직 추가하지 않습니다.

이유:

- hook 테스트에는 React 렌더링 테스트 도구가 필요합니다.
- `@testing-library/react` 또는 유사 도구를 추가해야 하므로 의존성 추가 결정이 필요합니다.
- 현재는 순수 유틸과 저장 데이터 보정 테스트만으로 회귀 위험을 먼저 낮춥니다.

## 3. React 테스트 도구 도입 기준

현재 결정: **아직 도입하지 않습니다.**

이유는 현재 회귀 위험이 순수 유틸/저장 데이터 보정 테스트로 어느 정도 커버되고 있고, React 테스트 도구를 추가하면 설정 파일과 테스트 작성 방식까지 함께 정해야 하기 때문입니다.

아래 중 하나가 실제로 필요해지면 React 테스트 도구 도입을 검토합니다.

- 퀘스트 완료, 수정, 삭제 흐름에서 반복 회귀가 생긴다.
- Supabase 동기화로 비동기 상태 전이가 복잡해진다.
- 알림 구독, 서비스워커, 권한 상태처럼 브라우저 API mock이 필요해진다.
- 수동 확인만으로 배포 전 검증 시간이 너무 길어진다.

권장 후보:

- `@testing-library/react`
- `@testing-library/user-event`
- `vitest`

도입 전에는 테스트 도구, 설정 파일, 스크립트 변경 범위를 별도 작업으로 제안합니다.

## 4. useQuests 테스트 후보

React 테스트 도구를 도입하면 아래 순서로 테스트합니다.

1. 완료되지 않은 퀘스트 완료 시 `completedToday=true`가 되고 완료 퀘스트가 반환된다.
2. 이미 완료된 퀘스트를 다시 완료해도 `null`을 반환하고 히스토리가 늘지 않는다.
3. 존재하지 않는 id를 완료해도 상태가 바뀌지 않는다.
4. 퀘스트 수정 시 제목, 카테고리, 난이도, 반복 주기만 바뀐다.
5. 카테고리 삭제 시 연결된 퀘스트는 남고 `categoryId:null`이 된다.

## 5. 빌드 검증

빌드는 여전히 사용자 PowerShell 기준으로 확인합니다.

```powershell
npm.cmd run build
```

Codex 세션에서 `spawn EPERM`이 나면 앱 코드 오류로 단정하지 않고, `docs/dev-environment.md`의 절차를 따릅니다.

## 6. AddQuestModal 분리 기준

현재 결정: **지금은 분리하지 않습니다.**

이유:

- 새 퀘스트, 추천 템플릿, 수정 모드가 한 컴포넌트에 있지만 현재 동작은 안정적입니다.
- 분리는 UI 구조 변경이므로 기능 작업과 섞이면 회귀 확인 범위가 커집니다.
- 다음 큰 기능이 붙을 때 필요한 부분만 작게 분리하는 편이 안전합니다.

분리 전 TODO:

- [ ] 이번 작업이 문구/스타일 수정인지, 입력 구조 변경인지 구분합니다.
- [ ] 새 필드가 퀘스트 생성과 수정 양쪽에 모두 필요한지 확인합니다.
- [ ] 추천 템플릿 흐름과 직접 만들기 흐름이 함께 바뀌는지 확인합니다.
- [ ] 카테고리 0개 안내와 미분류 정책에 영향이 있는지 확인합니다.
- [ ] 분리 후 수동 확인 범위가 커지는지 먼저 적습니다.

분리 시점:

- 추천 템플릿 편집/관리 기능이 추가된다.
- 미분류 탭을 정식으로 열어 퀘스트 추가/수정 정책이 바뀐다.
- 퀘스트 폼에 알림 시간, 메모, 목표량 같은 필드가 추가된다.
- AddQuestModal 수정 시 같은 조건문을 반복해서 고치게 된다.

분리 후보:

- `QuestFormFields`: 제목, 카테고리, 난이도, 반복 주기 입력
- `QuestTemplateList`: 추천 템플릿 필터와 목록
- `EmptyCategoryNotice`: 카테고리 0개 안내

분리 원칙:

- 한 번에 전부 나누지 않고, 변경이 필요한 영역 하나만 분리합니다.
- 새 컴포넌트는 기존 props 이름과 도메인 문자열을 그대로 사용합니다.
- 분리와 기능 변경을 같은 커밋에 크게 섞지 않습니다.
- 분리 후에는 새 퀘스트 생성, 추천 추가, 퀘스트 수정, 카테고리 0개 안내를 모두 확인합니다.

분리하지 않는 기준:

- 새 필드 추가 없이 문구나 스타일만 조금 바꾸는 작업
- 추천 템플릿 목록의 데이터만 바꾸는 작업
- 카테고리 0개 안내 문구만 조정하는 작업
