# Product Decisions

Quest의 다음 기능을 구현하기 전에 합의해 둔 제품 정책 메모입니다.
현재 MVP 동작을 유지하면서, 기능이 커지는 지점을 별도 작업으로 분리하기 위한 기준입니다.

## 1. 실제 푸시 발송

### 현재 범위

- 설정 화면은 알림 ON/OFF, 브라우저 권한 상태, 알림 시간을 저장합니다.
- 실제 푸시 발송은 아직 연결하지 않습니다.
- 저장된 시간은 향후 발송 로직이 사용할 사용자 설정값으로 봅니다.

### 선택지

| 방식 | 장점 | 한계 | 판단 |
|------|------|------|------|
| 로컬 브라우저 알림 | 서버 없이 가볍게 시작 가능 | 앱이 닫히면 안정적인 예약 발송이 어렵고, 모바일/PWA 지원 차이가 큼 | MVP에는 부적합 |
| Supabase + 서버 푸시 | 사용자별 구독 저장과 닫힌 앱 알림에 적합 | Supabase 인증/DB/Edge Function 설계가 필요 | v2 권장 |
| 별도 푸시 서버 | 가장 유연함 | 운영 부담과 보안 관리가 커짐 | 초기에는 보류 |

### 결정

실제 푸시 발송은 **Supabase 연동 정책을 정한 뒤 v2 기능으로 구현**합니다.
구조 초안은 [Supabase Push Notification Design](./supabase-push-design.md)에 둡니다.

구현 전에 아래를 먼저 결정합니다.

- 로그인 없이 기기 단위 구독만 저장할지, 사용자 계정 단위로 저장할지
- 알림 시간과 타임존을 어디에 저장할지
- iOS PWA, Android Chrome, 데스크톱 브라우저의 지원 차이를 어떻게 안내할지
- 사용자가 알림을 끈 경우 서버 구독을 삭제할지 비활성화만 할지

## 2. 긴 퀘스트 제목

### 현재 정책

- 퀘스트 제목은 한 줄 말줄임으로 표시합니다.
- 오른쪽에는 몬스터, 수정, 삭제 버튼을 고정 액션 영역으로 유지합니다.
- 긴 제목 때문에 카드가 답답해지는 실제 사용 사례가 쌓이기 전까지는 카드 높이를 늘리지 않습니다.

### 확인 기준

테스트용 긴 제목 5개를 추가해 모바일 폭에서 확인합니다.

- 제목 의미를 알아보기 어려울 정도로 잘리는가
- 카테고리, 난이도, XP 정보가 겹치거나 밀리는가
- 수정/삭제 버튼 터치 영역이 답답해지는가

### 조정 순서

1. 제목 2줄 허용을 먼저 검토합니다.
2. 그래도 답답하면 메타 정보를 더 작은 칩 구조로 정리합니다.
3. 마지막으로 카드 높이와 오른쪽 액션 영역 폭을 함께 조정합니다.

## 3. 미분류 퀘스트

### 현재 정책

- 새 퀘스트는 반드시 카테고리에 연결합니다.
- 카테고리가 0개이면 퀘스트 추가 대신 카테고리 생성 안내를 보여줍니다.
- 카테고리 삭제로 생긴 `categoryId: null`은 데이터 보존용 임시 상태입니다.
- 정식 `미분류` 탭은 아직 제공하지 않습니다.

### 정식 기능화 조건

아래 불편이 실제로 확인되면 별도 기능으로 설계합니다.

- 카테고리 삭제 후 남은 퀘스트를 찾기 어렵다.
- 사용자가 의도적으로 카테고리 없는 퀘스트를 만들고 싶어 한다.
- 전체 탭과 카테고리 통계에서 미분류 상태가 헷갈린다.

정식 기능으로 만들 때는 홈 탭, 퀘스트 추가/수정, 카테고리 통계, 추천 템플릿 동작을 함께 정합니다.
